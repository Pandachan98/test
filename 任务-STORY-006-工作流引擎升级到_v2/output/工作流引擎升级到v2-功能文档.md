## 1. 建设背景

当前平台需要将工作流能力从单点审批或模块内流程逻辑升级为可复用、可配置、可审计的 v2 工作流引擎。v2 引擎目标不是只新增一个审批页面，而是为项目经营协同平台内的任务、审批、协同、通知、审计等场景提供统一的流程运行底座。

升级后的工作流引擎应支持流程模型配置、版本发布、实例运行、任务处理、规则判断、异常干预、消息联动、过程审计和后续入库校验。对于已有业务模块，v2 需提供兼容策略，避免因为引擎升级导致历史流程无法查询、待办无法处理或审批结果丢失。

## 2. 建设目标

1. 统一流程模型：将不同业务模块中的流程配置抽象为流程定义、节点、连线、条件、处理策略和版本。
2. 支持版本化发布：流程配置必须经过草稿、校验、发布、停用、废弃等生命周期管理，避免直接修改线上流程。
3. 提供稳定运行时：支持流程实例创建、任务生成、节点推进、撤回、驳回、转办、委派、超时、终止等核心能力。
4. 增强规则能力：支持基于业务字段、组织、角色、金额、项目、标段、供应商、风险等级等条件进行流转判断。【待确认具体字段】
5. 完成审计追踪：每一次配置变更、实例推进、人工干预和系统回调均保留可追溯记录。
6. 支撑集成接入：为业务系统提供发起流程、查询状态、处理任务、监听事件、补偿重试等接口能力。【待研发确认接口协议】

## 3. 范围边界

### 3.1 本期建议纳入范围

| 模块 | 功能范围 | 说明 |
| --- | --- | --- |
| 流程定义管理 | 新建、编辑、复制、校验、发布、停用、废弃 | 面向流程管理员或业务配置员 |
| 流程版本管理 | 草稿版本、发布版本、历史版本、版本对比 | 线上实例默认绑定发起时版本 |
| 节点与连线配置 | 开始、审批、会签、并行、条件、自动、抄送、结束 | 节点能力按配置开放 |
| 条件规则 | 字段条件、组合逻辑、默认分支、规则预检 | 条件字段需从业务对象或表单映射 |
| 实例运行 | 发起、推进、挂起、恢复、撤回、驳回、取消、终止 | 运行时不可被流程定义直接覆盖 |
| 待办任务 | 认领、审批、转办、委派、加签、退回、评论、附件 | 具体操作受节点策略控制 |
| 消息通知 | 待办、抄送、催办、超时、流程完成、异常失败 | 通知渠道需确认 |
| 监控审计 | 实例轨迹、节点耗时、异常日志、配置变更日志 | 支撑运营和问题排查 |
| 对外集成 | API、事件回调、幂等、补偿、重试 | 与业务模块低耦合 |

### 3.2 本期不建议纳入范围

- 独立 BPMN 全量建模器的复杂高级能力，例如脚本任务、事务补偿建模、跨租户流程编排。【待确认是否有采购或替代方案】
- 组织架构、角色权限、用户中心的底层重建。v2 应复用现有组织与权限能力。
- 业务表单引擎全量重构。v2 仅定义表单字段映射和流程变量读取方式。
- 生产环境历史流程数据大规模自动迁移。建议先提供查询兼容和增量切换，迁移策略需专项评估。【待研发确认】

## 5. 核心功能

### 5.1 流程定义管理

流程定义是 v2 引擎的配置入口，用于描述某类业务流程的稳定身份、业务归属、适用范围和当前发布版本。

#### 5.1.1 新建流程定义

管理员可创建流程定义，需填写：

| 字段 | 说明 | 必填 |
| --- | --- | --- |
| 流程编码 | 全局唯一编码，用于 API 和业务模块绑定 | 是 |
| 流程名称 | 面向用户展示的名称 | 是 |
| 业务域 | 归属模块或业务线 | 是 |
| 业务对象 | 关联的单据或业务实体 | 是 |
| 适用范围 | 项目、组织、业务类型等过滤条件 | 否 |
| 发起权限 | 哪些角色或人员可发起 | 是 |
| 管理员 | 负责维护流程的用户或角色 | 是 |
| 备注 | 配置说明 | 否 |

#### 5.1.2 编辑与复制

- 未发布草稿可直接编辑。
- 已发布版本不可原地修改，只能基于发布版本复制出新草稿。
- 复制流程时应同时复制节点、连线、规则、通知和权限配置，并重新生成草稿版本号。
- 删除仅允许删除从未发布且无实例的草稿定义；已发布定义只能停用或废弃。

#### 5.1.3 配置校验

发布前必须完成结构校验：

- 必须存在且仅存在一个开始节点。
- 必须至少存在一个结束节点。
- 所有节点必须可从开始节点抵达。
- 条件分支必须有默认分支或完全覆盖条件。
- 处理人配置不能为空，除非节点类型为自动节点、抄送节点或结束节点。
- 会签、并签、或签规则必须明确通过阈值。
- 自动节点必须配置动作、接口或系统任务，并定义失败处理策略。

### 5.2 流程版本管理

v2 引擎必须将“流程定义”和“流程版本”分离，保证线上流程稳定可追溯。

| 状态 | 含义 | 允许操作 |
| --- | --- | --- |
| 草稿 | 尚未发布，可编辑 | 编辑、校验、提交发布、删除 |
| 待发布 | 已通过校验，等待发布确认 | 发布、撤回草稿 |
| 已发布 | 可被业务发起使用 | 查看、复制新版本、停用 |
| 已停用 | 不允许新发起，但历史实例继续运行 | 恢复、废弃、查看 |
| 已废弃 | 不再使用，仅保留审计 | 查看、复制 |

版本规则：

- 新实例发起时绑定当前发布版本。
- 已运行实例继续使用发起时绑定的版本，不随新版本自动变化。
- 停用流程不影响已运行实例，除非管理员执行专项终止或迁移。
- 版本发布需记录发布人、发布时间、发布说明、变更摘要和校验结果。

### 5.3 节点配置

#### 5.3.1 节点类型

| 节点类型 | 功能 | 关键配置 |
| --- | --- | --- |
| 开始节点 | 定义流程入口 | 发起权限、业务对象、表单映射 |
| 审批节点 | 单人或多人审批 | 处理人、审批动作、退回策略、超时策略 |
| 会签节点 | 多人共同处理 | 通过比例、全部通过、任一拒绝、顺序或并行 |
| 并行节点 | 同时生成多个分支任务 | 分支合并规则 |
| 条件节点 | 按规则进入不同分支 | 条件表达式、默认分支 |
| 抄送节点 | 通知相关人员 | 抄送对象、触发时机 |
| 自动节点 | 调用系统动作或接口 | 动作类型、入参映射、失败策略 |
| 计时节点 | 等待指定时间或业务时间点 | 时长、工作日规则、超时出口 |
| 子流程节点 | 发起另一个流程 | 子流程编码、父子流程关系 |
| 结束节点 | 结束流程 | 结果标记、回写规则 |

#### 5.3.2 处理人策略

处理人支持以下来源，具体接入优先级需研发确认：

1. 固定人员。
2. 固定角色。
3. 发起人上级。
4. 部门负责人。
5. 项目负责人。
6. 标段负责人。
7. 供应商联系人。【待确认是否纳入】
8. 表单字段指定人员。
9. 外部接口返回人员。【待研发确认】

处理人为空时的兜底策略：

- 阻断流程并进入异常队列。
- 转交流程管理员。
- 使用默认角色兜底。

建议默认采用“阻断流程并进入异常队列”，避免错误审批。

#### 5.3.3 审批动作

| 动作 | 说明 | 影响 |
| --- | --- | --- |
| 同意 | 当前任务通过 | 推进到下一节点或等待其他并行任务 |
| 驳回 | 拒绝当前流程 | 可结束流程或退回指定节点 |
| 退回 | 返回上一节点或指定节点 | 保留退回记录 |
| 撤回 | 发起人收回未完成流程 | 需满足节点允许撤回 |
| 转办 | 当前处理权转给他人 | 原处理人不再处理 |
| 委派 | 委托他人处理后回到本人 | 是否回到本人由节点配置决定 |
| 加签 | 临时增加处理人 | 支持前加签、后加签、并加签【待确认】 |
| 终止 | 管理员或业务规则强制结束 | 必须记录终止原因 |

### 5.4 条件规则引擎

条件规则用于决定流程分支、节点跳转、处理人选择和通知策略。

规则表达应支持：

- 等于、不等于、包含、不包含。
- 大于、小于、区间。
- 为空、不为空。
- 属于组织、属于角色、属于项目。
- 多条件 AND / OR 组合。
- 默认分支兜底。

规则变量来源：

| 来源 | 示例 | 说明 |
| --- | --- | --- |
| 表单字段 | 金额、合同类型、风险等级 | 由业务表单映射 |
| 业务对象 | 项目、标段、供应商、合同 | 由业务模块传入 |
| 组织用户 | 部门、岗位、角色、上级 | 复用组织权限服务 |
| 系统上下文 | 发起时间、发起人、流程版本 | 引擎内置 |

规则预检：

- 发布前校验字段是否存在。
- 发布前校验枚举值是否有效。
- 发布前校验是否存在不可达分支。
- 运行时若规则计算失败，实例进入异常队列，不应默认放行。

### 5.5 流程实例运行

流程实例是一次具体业务流程的运行记录。

#### 5.5.1 发起流程

业务模块通过页面或接口发起流程时，需要传入：

| 参数 | 说明 | 必填 |
| --- | --- | --- |
| workflowCode | 流程编码 | 是 |
| businessKey | 业务单据唯一键 | 是 |
| businessTitle | 业务标题 | 是 |
| initiatorId | 发起人 | 是 |
| businessPayload | 业务字段快照 | 是 |
| callbackUrl | 结果回调地址 | 否 |
| idempotencyKey | 幂等键 | 是 |

发起规则：

- 同一 `workflowCode + businessKey + idempotencyKey` 不得重复创建实例。
- 发起时应复制业务字段快照，用于流程审计和规则计算。
- 发起失败需返回明确错误码和失败原因。

#### 5.5.2 实例状态

| 状态 | 含义 | 可见动作 |
| --- | --- | --- |
| 初始化中 | 实例正在创建 | 仅系统可见 |
| 运行中 | 流程正常推进 | 查看、催办、处理 |
| 等待中 | 等待计时、外部回调或子流程 | 查看、补偿 |
| 已挂起 | 暂停推进 | 恢复、终止 |
| 已完成 | 正常结束 | 查看轨迹 |
| 已驳回 | 被审批拒绝 | 查看、重新发起【待确认】 |
| 已取消 | 发起人或业务方取消 | 查看 |
| 已终止 | 管理员强制结束 | 查看审计 |
| 异常 | 规则、接口或数据异常 | 重试、跳过、终止、修复 |

### 5.6 待办任务

待办任务面向处理人展示，需支持列表、详情、处理、批量操作和历史查询。

#### 5.6.1 待办列表字段

| 字段 | 说明 |
| --- | --- |
| 任务标题 | 业务标题或节点标题 |
| 流程名称 | 所属流程 |
| 当前节点 | 当前处理节点 |
| 发起人 | 流程发起人 |
| 发起时间 | 实例创建时间 |
| 到达时间 | 当前任务生成时间 |
| 截止时间 | 超时计算结果 |
| 优先级 | 普通、紧急、超期等【待确认】 |
| 状态 | 待处理、已处理、已转办、已取消 |

#### 5.6.2 任务详情

任务详情应包含：

- 业务摘要。
- 表单字段快照。
- 当前节点说明。
- 可执行操作。
- 审批意见输入。
- 附件上传。
- 历史流转轨迹。
- 相关抄送和评论。

### 5.7 催办与超时

v2 应将催办和超时策略配置化。

| 能力 | 说明 |
| --- | --- |
| 自动催办 | 到达指定时长后自动提醒处理人 |
| 手动催办 | 发起人或管理员手动提醒 |
| 多级升级 | 超时后提醒上级或流程管理员 |
| 超时转办 | 超时后转交指定角色或人员 |
| 超时跳转 | 超时后进入指定节点【待确认是否允许】 |
| 工作日规则 | 支持自然日、工作日和节假日配置【待研发确认】 |

建议所有催办和超时动作写入流程轨迹，避免用户感知与审计记录不一致。

### 5.8 异常处理

异常来源包括：

- 处理人解析为空。
- 条件规则计算失败。
- 自动节点接口调用失败。
- 回调业务系统失败。
- 并行合并状态不一致。
- 历史版本配置缺失。
- 业务对象被删除或不可访问。

异常处理方式：

| 方式 | 使用场景 | 审计要求 |
| --- | --- | --- |
| 重试 | 接口临时失败 | 记录重试次数和结果 |
| 修复变量后重试 | 业务字段或组织数据错误 | 记录修改人和修改前后值 |
| 跳过节点 | 非关键自动节点失败 | 记录跳过原因和审批人 |
| 转人工 | 自动节点无法继续 | 生成管理员任务 |
| 终止实例 | 数据不可恢复 | 记录终止原因和影响范围 |

## 6. 关键业务流程

### 6.1 流程配置发布流程

1. 流程管理员创建或复制流程草稿。
2. 配置节点、连线、条件、处理人、通知和异常策略。
3. 系统执行结构校验和规则预检。
4. 校验通过后进入待发布。
5. 发布人填写发布说明和变更摘要。
6. 系统生成发布版本并记录版本快照。
7. 新发起实例使用最新已发布版本。

### 6.2 业务流程发起流程

1. 业务模块提交流程发起请求。
2. 引擎校验流程定义是否存在、已发布、未停用。
3. 引擎校验发起人权限。
4. 引擎保存业务字段快照。
5. 引擎创建流程实例。
6. 引擎计算第一个处理节点。
7. 引擎生成待办任务和通知。
8. 引擎返回实例编号和当前状态。

### 6.3 任务处理流程

1. 处理人打开待办任务。
2. 系统校验任务是否仍处于可处理状态。
3. 处理人选择动作并填写意见。
4. 系统保存处理记录。
5. 系统根据节点策略推进流程。
6. 若进入下一节点，生成新的待办或自动任务。
7. 若流程结束，回写业务结果并触发完成通知。

### 6.4 异常修复流程

1. 实例进入异常队列。
2. 系统记录异常类型、节点、上下文和错误详情。
3. 运维或流程管理员查看异常。
4. 管理员选择重试、修复变量、转人工、跳过或终止。
5. 系统记录干预动作和原因。
6. 流程恢复推进或进入终态。

## 7. 数据对象

| 对象 | 说明 | 关键字段 |
| --- | --- | --- |
| WorkflowDefinition | 流程定义 | code、name、businessDomain、businessObject、status |
| WorkflowVersion | 流程版本 | definitionCode、version、status、publishedAt、snapshot |
| WorkflowNode | 节点定义 | nodeId、type、name、handlerPolicy、timeoutPolicy |
| WorkflowTransition | 连线定义 | fromNode、toNode、condition、priority |
| WorkflowInstance | 流程实例 | instanceId、businessKey、version、status、initiator |
| WorkflowTask | 待办任务 | taskId、instanceId、nodeId、assignee、status、dueAt |
| WorkflowActionLog | 处理记录 | action、operator、comment、attachments、createdAt |
| WorkflowEvent | 事件记录 | eventType、payload、retryCount、status |
| WorkflowAuditLog | 审计日志 | targetType、targetId、operation、before、after |

## 8. 权限控制

权限应分为配置权限、运行权限、查看权限和运维权限。

| 权限 | 说明 |
| --- | --- |
| 流程定义管理 | 新建、编辑、复制、发布、停用流程 |
| 版本发布 | 将草稿发布为线上版本 |
| 流程发起 | 在指定业务域发起流程 |
| 任务处理 | 处理分配给自己的任务 |
| 流程查看 | 查看本人发起、本人处理、被抄送或授权范围内流程 |
| 异常干预 | 对异常实例执行重试、跳过、终止等动作 |
| 审计查看 | 查看配置变更和流程轨迹 |

安全要求：

- 处理任务时必须二次校验处理人身份。
- 转办、委派、加签必须校验目标人员是否合法。
- 管理员干预必须记录原因。
- 接口调用需支持签名、鉴权和幂等。【待研发确认】

## 9. 集成能力

### 9.1 API 能力

建议提供以下接口，最终路径和协议待研发确认：

| 接口 | 功能 |
| --- | --- |
| 创建流程实例 | 业务模块发起流程 |
| 查询流程实例 | 查询实例状态、当前节点、处理人 |
| 取消流程实例 | 业务模块取消流程 |
| 完成任务 | 处理人提交审批动作 |
| 查询待办 | 获取用户待办任务 |
| 查询流程轨迹 | 获取节点流转记录 |
| 注册回调 | 配置业务系统回调地址 |
| 事件重试 | 对失败回调执行补偿 |

### 9.2 事件能力

v2 应支持事件发布：

- instance.created
- task.created
- task.completed
- instance.completed
- instance.rejected
- instance.canceled
- instance.failed
- definition.published

事件要求：

- 每个事件具备唯一 eventId。
- 回调失败需要重试和死信记录。
- 业务方接收事件需要幂等处理。
- 敏感字段需要脱敏或授权后传输。

## 11. 验收标准

### 11.1 功能验收

- 能创建流程定义并保存草稿。
- 能完成发布前结构校验。
- 能发布新版本并保留版本快照。
- 新实例能绑定发布版本并生成首个任务。
- 处理人能完成同意、驳回、退回、转办等动作。
- 条件节点能按业务字段进入不同分支。
- 会签节点能按阈值正确推进。
- 自动节点失败后能进入异常队列。
- 管理员可查看实例轨迹和审计日志。
- 所有关键动作都有操作人、时间、结果和原因记录。

### 11.2 兼容验收

- 已有历史流程可查询。
- v1 存量待办不因 v2 发布而丢失。【待研发确认迁移策略】
- 新发起流程使用 v2 发布版本。
- 业务模块可通过配置切换是否启用 v2。
