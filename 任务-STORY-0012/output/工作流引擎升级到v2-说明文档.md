## 2. 背景与问题

现有工作流能力需要升级到 v2，以支撑更复杂的业务审批、任务流转、异常处理和审计追踪。结合丰匠 AI 作业治理 Demo 的任务背景，v2 升级应重点解决以下问题：

1. 流程定义、流程实例、节点执行和任务待办之间缺少统一模型，后续扩展成本高。
2. 流程版本变更后，历史实例、在途实例和新发起实例的兼容规则需要明确。
3. 节点超时、驳回、撤回、重试、跳过、人工干预等异常分支需要可配置、可追踪。
4. 外部业务系统接入工作流时，需要明确事件、回调、状态同步和失败补偿机制。
5. 关键流程动作需要具备审计证据，支撑后续入库校验、过程复盘和责任追踪。

## 3. 建设目标

### 3.1 业务目标

- 支持业务方通过统一工作流能力编排审批、协同、派单、复核、归档等流程。
- 支持同一业务对象在不同流程版本下稳定运行，避免流程升级影响历史数据。
- 支持业务用户在待办中心、流程详情和业务单据中查看当前节点、处理人、状态和操作记录。
- 支持运营或管理员对流程定义、版本发布、实例状态和异常任务进行可控管理。

### 3.2 技术目标

- 建立 v2 流程定义模型，区分流程模板、流程版本、节点、边、条件和动作。
- 建立流程实例运行时模型，记录实例状态、节点状态、任务状态、变量快照和事件日志。
- 提供标准事件机制，支持业务系统监听流程启动、节点完成、流程完成、异常终止等事件。
- 提供审计日志和运行证据，支持后续与 AI 作业治理链路的凭证登记、hash 校验、复盘报告对齐。

### 3.3 非目标范围

- 不在本次说明文档中确定具体 UI 高保真页面。
- 不在本次说明文档中锁定最终数据库表结构。
- 不在本次说明文档中承诺具体审批组织、角色、阈值或业务状态映射。
- 不替代真实生产 PRD、接口设计文档、测试用例或上线方案。

## 5. v2 核心能力说明

### 5.1 流程定义与版本管理

工作流 v2 需要将“流程模板”和“流程版本”拆分管理。

- 流程模板用于表达业务流程的长期身份，例如“合同审批流程”“采购申请流程”。
- 流程版本用于表达某次已发布的流程结构，例如 `v1.0`、`v1.1`、`v2.0`。
- 每次发布版本后，需要固化节点、边、条件、处理人规则和表单映射。
- 新实例默认使用最新启用版本，历史实例继续绑定发起时版本。
- 在途实例是否允许迁移到新版本需要按业务场景控制，默认建议不自动迁移。

待确认事项：

- 【待确认】版本号规则是否使用语义化版本，还是使用系统自增版本。
- 【待研发确认】流程定义是否需要支持可视化拖拽配置。
- 【待确认】在途实例迁移是否纳入 v2 首期范围。

### 5.2 节点与流转规则

v2 节点模型建议至少支持以下类型：

- 开始节点：流程实例入口。
- 人工处理节点：生成待办任务，由指定处理人完成。
- 自动服务节点：调用外部服务或系统内部能力自动执行。
- 条件网关：根据变量、业务字段或处理结果选择分支。
- 并行网关：支持多分支并发执行及汇聚。
- 结束节点：标记流程完成、终止或取消。

节点配置建议包含：

- 节点编码、节点名称、节点类型。
- 处理人规则：固定人员、角色、部门负责人、发起人上级、业务对象负责人等【待确认】。
- 表单权限：字段可见、可编辑、必填、只读。
- 动作集合：通过、驳回、转交、加签、撤回、终止、暂存【待确认】。
- 超时规则：超时提醒、自动升级、自动流转或转人工处理【待补充真实运营数据】。

### 5.3 流程实例运行

流程实例是某个业务对象发起后产生的运行记录。v2 应明确实例生命周期：

| 状态 | 含义 | 典型触发 |
| --- | --- | --- |
| `draft` | 草稿，尚未正式发起 | 发起人保存草稿 |
| `running` | 运行中 | 发起流程成功 |
| `suspended` | 暂停 | 管理员或系统异常暂停 |
| `completed` | 正常完成 | 到达结束节点 |
| `rejected` | 被驳回到发起人或指定节点 | 节点处理人驳回 |
| `cancelled` | 已撤回或取消 | 发起人撤回、管理员取消 |
| `terminated` | 异常终止 | 系统异常或人工终止 |

实例运行时需要保存：

- 绑定的流程模板和流程版本。
- 业务对象类型和业务对象 ID。
- 当前节点集合。
- 当前处理人和待办任务。
- 流程变量快照。
- 节点进入、离开、驳回、重试、终止等事件。

### 5.4 待办与处理任务

人工处理节点进入后，系统应生成待办任务。待办任务需要支持：

- 待办创建、领取、处理、转交、关闭。
- 与业务单据详情联动，用户可从业务单据看到流程状态，也可从待办进入单据处理。
- 处理意见、附件、操作时间、处理结果留痕。
- 多处理人场景下的会签、或签、顺序签规则【待确认】。
- 移动端或 PC 端入口范围【待确认】。

建议首期至少支持单节点单处理人或角色处理人，复杂会签/加签规则可作为二期扩展，除非真实业务已明确要求首期上线。

### 5.5 异常、补偿与人工干预

工作流 v2 需要将异常从“隐性失败”转为“可见、可追踪、可处理”。

建议支持的异常类型：

- 外部接口调用失败。
- 节点处理人缺失。
- 条件表达式无法计算。
- 待办超时未处理。
- 数据权限或业务对象不存在。
- 流程定义已禁用或版本不兼容。

建议支持的处理动作：

- 重试自动节点。
- 指派新的处理人。
- 暂停流程实例。
- 跳过当前节点【待研发确认】。
- 回退到指定节点【待研发确认】。
- 终止实例并记录原因。

### 5.6 审计与治理证据

v2 升级需要保留关键审计信息，至少包括：

- 流程定义创建、修改、发布、停用记录。
- 节点配置变更记录。
- 实例启动、节点进入、节点完成、驳回、撤回、终止记录。
- 处理人、处理动作、处理意见、处理时间、来源端。
- 自动节点调用入参摘要、出参摘要、错误码和重试次数。

与 AI 作业治理 Demo 对齐时，正式产物和审计文件需要进入 `trace/run_certificate.json` 并计算 SHA256，确保签发后文件内容不可静默修改。

## 6. 数据对象建议

| 对象 | 用途 | 关键字段建议 |

## 7. 接口与事件建议

### 7.1 基础接口

- 创建流程模板。
- 发布流程版本。
- 启动流程实例。
- 查询流程实例状态。
- 查询业务对象关联流程。
- 查询用户待办。
- 处理待办任务。
- 撤回或终止流程实例。
- 查询流程运行轨迹。

### 7.2 事件通知

建议对外提供以下事件：

- `workflow.instance.started`
- `workflow.node.entered`
- `workflow.task.created`
- `workflow.task.completed`
- `workflow.instance.completed`
- `workflow.instance.rejected`
- `workflow.instance.cancelled`
- `workflow.instance.terminated`
- `workflow.exception.raised`

事件投递方式、幂等键、重试次数、死信队列和签名机制均为 `【待研发确认】`。

## 8. 权限与安全

- 流程模板配置权限应限制在流程管理员或系统管理员范围内。
- 流程版本发布需要记录发布人、发布时间和发布说明。
- 处理人只能查看并处理自己有权限的待办。
- 管理员人工干预应强制填写原因并写入审计日志。
- 关键操作需要记录来源端、操作 IP 或设备标识【待研发确认】。
- 涉及敏感字段的表单权限需要和业务系统权限模型一致【待确认】。

## 9. 迁移与兼容

v2 升级建议按以下方式控制风险：

1. 先完成 v2 模型与运行时能力，不直接替换全部存量流程。
2. 选择低风险业务流程试点，验证版本发布、实例运行、待办处理、异常恢复。
3. 对存量流程进行映射评估，区分可平滑迁移、需重新配置、暂不迁移三类。
4. 在切换前保留 v1 查询和历史实例查看能力。
5. v2 上线后保留回滚方案，避免流程引擎异常影响核心业务办理。
